Learning Backbone.JS in 2017

About

Why would anyone want to learn Backbone when there are many newer, faster, cooler JS frameworks? Well, first off, Backbone is not a framework. Second, I am not learning this for money. I am learning Backbone because it is the next step in the progress to understand Web Development.

There is more than one way to do it.

KISS.js

Keep it Simple, Silly

I am part of the hacker school genearation. I went to a 12 week bootcamp that attempted to teach me all there is to know. Since I was one of the first types of people to attend a bootcamp, it was short and full of holes. I left that place with nothing but the happy path. The happy path does not exist in the real world. You must venture into topics, layers, code, languages that were not discussed in Code Class. It is extremely difficult bordering on the state of impossibility to learn everything at the same time while trying to apply it. So now that I'm 5 years in and growing with facination about computers know that I am not worried about getting fired, I'm starting over and taking the scenic reoute.

All of this preface should really go into a different blog, but it is the reason why I'm learning backbone in 2017. This is the time and year that I'm finally understanding why it was brought onto the scene.

From there to here.

To put it into context... I read the Definitive guide to JavaScript, twice. I read JS the Good Parts, twice. I read JavaScript Patterns, twice. I committed to learning ES2015.

'JQuery is for lazy people.'

I am embaressed to say that all my blog reading and surveying of what to learn led me to initially believe the above about JQuery. I was such a noob that I looked down on JQuery because I was using Angular and JQuery was old school. But now, I've come to realize what a moron I was and I have studied JQuery and see why it is an essential tool that I plan to keep as one of my non-negotiables. (That or Zepto) JQuery managed the DOM because JavaScript didn't realize that was a thing. The DOM needs a tool to query it and there is no better tool than JQuery or one that follows JQuery's lead.

Underscoring the importance of the right tool for the job

JQuery's popularity has been in part of handling all the things needed to be done in the Browser. Query, Traversing, AJAX, Data, Events, and more. The are several downsides to handling all these tasks. For example, difficulty in testing and breaking SOLID Principals, which I am into. Specifically the big one, Single Responsibility.

While JavaScript has many types of iterators, they fall short with other aspects of the language, like scope. Underscore provides a specific set of utility functions that are not avaialble in JQuery that solve some warty aspects of JavaScript. It is another hole that has been filled by a minial dependency. It also has a consistent API that can handle many of the uglier parts of writing JS code. After learning the JavaScript way first, I see the benefits that Underscore brings to the table and I wholeheartedly adopt the library as a personal dependency.

Do not serve me

Keeping things simple and focusing on the lessons at hand means keeping my mental workspace clean. JavaScript, or ECMASCRIPT 2015, or ES15 does not need to be interpreted. Meaning, it does not need a server. It does not need a build tool. It does not need anything to work in a modern browser. The same can be said about JQuery, Underscore, and Backbone.

Follow this logic and tell me what conclusion you would arrive if it is different. I want to learn about JavaScript on a Browser. I don't want to learn about servers, interpreters, build tools, or any other fancy tool to support JavaScript.

I have already taken the time to learn JavaScript, JQuery, and Underscore. Those are the only three requirements for me to learn Backbone.

Start with simple tools and add complexity as see fit.

Backbone's official documentation touts itself as filling a need that JQuery and Underscore do not meet. Okay, what is it? The good news is it is a small library that can be read in a single sitting if you were that into pain. Backbone has only two dependencies JQuery and Underscore. Backbone does not need to be interpreted.

The Problems Backbone solves

If you got with me this far, hopefully you see that this rant is not a case for why Backbone is better than other JS frameworks. Backbone addresses real ES15 issues by adding the Saw to your Hammer and Drill.

The third side of the trilogy.

The benefit of coming into web development late is there are many mentors and resources based on real world problems. Now that I have a solid grasp of what roles JQuery and Underscore provide, I am able to look at the third part of the trilogy.

Single Page Applicatoins (SPAs)seem to be the reason Backbone was written but my interest in Backbone is not synonymous with building one. Sticking with the right tool for the job, I wanted to understand what problem bakbone was solving. Well, it turns out that javascript heavy development ends up having many aspects of the 'page' interact with one another. That page interaction is an event driven process that updates data. Events can be fired by the server, browser, or user.

Up to this point, all events were managed in line with the activity they were associated with which resulted in lots of tight coupling and code duplication.

To make matters worse, the DOM began to have yet another responsibilty of housing important content, or state. The DOM was becoming stateful on a stateless platform.

One of the ways I've heard people tout JS is that it treats functions as first class citizens. Well, Backbone does the same thing but with events and data which up to this point is tangled with the DOM and JQuery. Backbone addresses these pain points (that are not addressed by JS, JQuery, or Underscore) with a simple, set of vanilla JavaScript conventions that solve these issues.

Let me summarize

As knowledge workers, we need to have knowledge of both the problems we face and our solutions to provide. In software, solutions come in the form of understanding our domain. I have chosen to be a full stack web developer. As such, that means I want to know the Browser, which is inseprable from JavaScript. JavaScript has the unfortunate history of not being a well designed language. However, that does not negate the fact that it is responsible for dealing with the DOM, Events, Data, and Servers.

I am taking the time to learn the web environment and all of it's responsibilities, challenges, and solutions so that I can truly master my craft.
